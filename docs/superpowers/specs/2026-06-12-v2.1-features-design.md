# ShortyQ v2.1 — Feature Expansion (Design)

**Date:** 2026-06-12
**Status:** Approved
**Builds on:** 2026-06-12-quantum-safe-rewrite-design.md (v2.0.0)

## Goal

Add eight features on top of the v2.0 ML-KEM-768 + AES-256-GCM architecture,
fully backward compatible with v2.0 payloads (minor version: 2.1.0):

1. Encrypted expiry
2. Encrypted metadata
3. Password-protected links
4. Key rotation
5. Deterministic short codes
6. Vanity short codes
7. Batch API
8. CLI keygen

## Design

### 1+2. Encrypted envelope (expiry + metadata)

Instead of encrypting the bare URL, encrypt a compact JSON envelope:

```json
{ "u": "<url>", "e": 1765497600000, "m": { ...arbitrary JSON... } }
```

`e` (epoch millis) and `m` are present only when the caller set
`expiresAt` / `metadata`.

**Decrypt fallback:** if the decrypted plaintext parses as a JSON object with
a string `u` property, treat it as an envelope; otherwise treat the plaintext
as a v2.0 bare URL. (A bare URL is never valid JSON, so the dispatch is
unambiguous.)

**Expiry enforcement:** `decryptUrl` and `decryptPayload` return `null` when
`e` is present and `e <= Date.now()`. The timestamp lives inside the
authenticated ciphertext, so it cannot be altered by whoever holds the
database.

**New function:**

```typescript
decryptPayload(payload, secretKey | secretKey[], options?)
  -> { url: string; metadata?: unknown; expiresAt?: Date } | null
```

`decryptUrl` keeps its `string | null` contract and delegates to the same
internals.

### 3. Password-protected links

`createShortUrl(url, { password })`:

- Generate a random 16-byte `kdfSalt`, added to the payload (base64).
- AES key = `sha256(sharedSecret || scrypt(password, kdfSalt, { N: 2**15, r: 8, p: 1, dkLen: 32 }))`
  instead of the raw shared secret. `scrypt` and `sha256` come from
  `@noble/hashes` (already a transitive dependency; becomes direct).
- Decryption: `decryptUrl(payload, secretKey, { password })`. Wrong or missing
  password -> GCM auth failure -> `null`.
- No `kdfSalt` field in payload -> passwordless path, byte-identical key
  derivation to v2.0 (raw sharedSecret).
- Supplying `options.password` for a payload without `kdfSalt` returns `null`
  (explicitly inconsistent inputs fail closed).

### 4. Key rotation

- Payloads gain `keyId`: base64 of the first 8 bytes of
  `sha256(publicKeyBytes)`. Always written by v2.1.
- `decryptUrl` / `decryptPayload` accept `secretKey: string | string[]`.
  With an array, try each key in order; wrong keys fail GCM auth safely.
  `keyId` is advisory (lets apps index keys); the library does not require it.
- New helper `getKeyId(publicKey: string): string` so apps can label their
  keys.
- v2.0 payloads (no `keyId`) decrypt fine with single key or array.

### 5. Deterministic short codes

- `generateKeyPair()` returns a third field `codeKey` (32 random bytes,
  base64) — additive, non-breaking for v2.0 consumers.
- `new ShortyQ({ publicKey, codeKey? })`;
  `createShortUrl(url, { deterministic: true })` computes the code as
  HMAC-SHA256(codeKey, url), base64url-encoded, truncated to `urlLength`.
- Same URL + same codeKey + same urlLength -> same code (dedupe across
  instances).
- Throws `"Deterministic codes require a codeKey"` if used without `codeKey`.
- Opt-in only; README documents that it reveals URL-equality.

### 6. Vanity short codes

- `createShortUrl(url, { shortCode: "summer-sale" })`.
- Validation: `/^[A-Za-z0-9_-]{4,100}$/` (nanoid alphabet, existing length
  bounds). Invalid -> throw `"Invalid custom short code"`.
- `shortCode` + `deterministic: true` together -> throw
  `"Cannot combine a custom short code with deterministic mode"`.
- Collision handling remains the caller's job.

### 7. Batch API

```typescript
shortyQ.createShortUrls(
  items: Array<string | { url: string; options?: CreateShortUrlOptions }>
) -> Array<{ shortCode: string; payload: EncryptedPayload }>
```

All-or-nothing: validates/throws on the first invalid item, no partial
results returned.

### 8. CLI keygen

- `src/cli.ts` with `#!/usr/bin/env node` shebang, compiled to `dist/cli.js`.
- `package.json`: `"bin": { "shortyq": "dist/cli.js" }`.
- `npx shortyq keygen` prints:

```
SHORTYQ_PUBLIC_KEY=<base64>
SHORTYQ_SECRET_KEY=<base64>
SHORTYQ_CODE_KEY=<base64>
```

- `npx shortyq keygen --json` prints the KeyPair as JSON.
- Any other invocation prints usage to stderr and exits 1.
- No CLI framework; plain `process.argv`.

### Unified options

```typescript
interface CreateShortUrlOptions {
  expiresAt?: Date | number;   // Date or epoch millis
  metadata?: unknown;          // JSON-serializable
  password?: string;
  shortCode?: string;          // vanity
  deterministic?: boolean;     // requires instance codeKey
}

interface DecryptOptions {
  password?: string;
}
```

All combinable except `shortCode` + `deterministic`.

`expiresAt` validation: must be a future date at creation time, else throw
`"expiresAt must be in the future"`. `metadata` must survive
`JSON.stringify` (functions/undefined/circular -> throw
`"Metadata must be JSON-serializable"`).

### Payload shape (v2.1)

```typescript
interface EncryptedPayload {
  kemCiphertext: string; // base64, 1088 bytes
  nonce: string;         // base64, 12 bytes
  ciphertext: string;    // base64, AES-256-GCM ct+tag
  keyId?: string;        // base64, 8 bytes — absent in v2.0 payloads
  kdfSalt?: string;      // base64, 16 bytes — present only for password links
}
```

### Compatibility matrix

| Producer | Consumer | Result |
|----------|----------|--------|
| v2.0 payload | v2.1 decrypt | works (bare-URL fallback, no kdfSalt -> raw key derivation) |
| v2.1 payload | v2.1 decrypt | works (envelope path) |
| v2.1 payload | v2.0 decrypt | decrypts, but returns the JSON envelope string rather than the URL |

Note: v2.1 always writes the envelope (`{"u": ...}`) so that expiry/metadata
handling is uniform. Consequence (third row): records written by v2.1 are not
cleanly readable by the v2.0 library. This is acceptable for a minor version
because the documented direction of compatibility is "new library reads old
data," and the npm package is upgraded as a unit. README documents this.

### File structure

- `src/index.ts` — public API (class, decrypt functions, generateKeyPair,
  getKeyId). Grows moderately.
- `src/cli.ts` — CLI entry point (keygen only).
- `src/index.test.ts` — extended suite.
- `src/cli.test.ts` — CLI smoke tests (spawn `node dist/cli.js` is overkill;
  test the exported `runCli(argv)` function directly, with the shebang file
  calling it).

### Dependencies

- Add direct dependency `@noble/hashes@^1.8.0` (already installed
  transitively via @noble/post-quantum): `sha256`, `hmac`, `scrypt`.
- No other new dependencies.

### Tests (beyond per-feature TDD)

- v2.0-shaped payload (bare URL, no keyId/kdfSalt) decrypts under v2.1
- Expired link -> null from both decryptUrl and decryptPayload
- Wrong password -> null; missing password -> null; password on
  passwordless payload -> null
- Combined: password + metadata + expiry round-trip
- Deterministic: stable across two ShortyQ instances with the same codeKey;
  different codeKey -> different code
- Rotation: array of [wrongKey, rightKey] decrypts; all-wrong -> null
- getKeyId(publicKey) matches payload.keyId
- Batch: mixed string/object items; first invalid item throws
- Vanity: valid accepted verbatim; invalid charset/length throws; combination
  with deterministic throws
- CLI: keygen output parses, keys decode to correct lengths; --json parses;
  unknown command exits non-zero
