# ShortyQ v2.2 — Hybrid X25519 + ML-KEM-768 Key Exchange (Design)

**Date:** 2026-06-12
**Status:** Approved
**Builds on:** v2.1 (2026-06-12-v2.1-features-design.md)

## Goal

Match modern OpenSSH's conservative posture (`mlkem768x25519`): combine
ML-KEM-768 with classical X25519 so confidentiality survives a break of
*either* algorithm. New keys are hybrid by default; all v2.0/v2.1 keys and
payloads keep working. Minor version: 2.2.0.

## Design

### Key format (concatenated bytes, base64)

| Key | v2.1 (legacy) | v2.2 (hybrid) |
|-----|---------------|----------------|
| publicKey | 1184 (ML-KEM ek) | 1216 = ML-KEM ek (1184) ‖ X25519 pk (32) |
| secretKey | 2400 (ML-KEM dk) | 2432 = ML-KEM dk (2400) ‖ X25519 sk (32) |

Mode is detected by length everywhere (constructor, getKeyId, tryDecrypt).
`generateKeyPair()` now returns hybrid keys. Legacy-size keys remain fully
accepted and produce/decrypt pure-ML-KEM payloads exactly as in v2.1.

### Payload

Hybrid payloads gain one field; its presence signals hybrid mode:

```typescript
interface EncryptedPayload {
  // ... existing fields ...
  /** Base64 X25519 ephemeral public key (32 bytes) — hybrid payloads only */
  x25519Ciphertext?: string;
}
```

### Combiner (encrypt with hybrid public key)

1. `(ssM, ctM) = ML-KEM-768.encapsulate(mlkemPk)`
2. ephemeral X25519 keypair `(eSk, ePk)`; `ssX = X25519(eSk, x25519Pk)`;
   `ctX = ePk`
3. Base AES key (dual-PRF pattern, transcript-bound like X-Wing):

```
baseKey = sha256( utf8("shortyq-hybrid-v1") || ssM || ssX || ctX || x25519Pk )
```

4. The existing password path composes on top: with a password,
   `aesKey = sha256(baseKey || scrypt(password, kdfSalt))` (i.e. baseKey
   replaces the raw sharedSecret as deriveAesKey's first input). Without,
   `aesKey = baseKey`.

We deliberately do NOT claim X-Wing compliance (different label/hash); the
combiner is documented as ShortyQ's own, following the standard
concatenate-and-hash construction used by TLS/SSH hybrids.

### Decryption

- Payload has `x25519Ciphertext` (decodes to 32 bytes, else null) → requires
  a 2432-byte hybrid secret key (legacy key → null); recompute
  `ssX = X25519(x25519Sk, ctX)` and the combiner.
- No `x25519Ciphertext` → v2.1 pure path; a hybrid secret key's ML-KEM
  prefix is used, so hybrid keys can also decrypt pure payloads encrypted
  to their ML-KEM half.
- Wrong/mismatched anything → GCM auth failure → null (try-each rotation
  arrays unchanged, may mix legacy and hybrid keys).

### keyId

`getKeyId` and constructor stamping: sha256 over the *full* public key bytes
(1184 or 1216), first 8 bytes, base64 — unchanged logic, both sizes accepted.

### Dependency

`@noble/curves@^1.9.0` (dual CJS/ESM, same family/author as existing deps);
`x25519` from `@noble/curves/ed25519`.

### Out of scope

No payload-format version bump; no changes to envelope, codes, batch, CLI
semantics (CLI prints hybrid keys automatically; its size assertions update).

### Tests

- Hybrid keypair sizes (1216/2432); CLI test sizes updated
- Hybrid round-trip (plain, password, metadata+expiry composed)
- Legacy keys still produce/decrypt pure payloads (v2.1 behavior pinned)
- Hybrid secret key decrypts a pure payload made for its ML-KEM half
- Pure (legacy) secret key returns null for hybrid payloads
- Tampered `x25519Ciphertext` → null; wrong-size x25519Ciphertext → null
- Rotation array mixing legacy + hybrid keys finds the right one
- getKeyId accepts both sizes, rejects others
